# PRD — 공통 규격 & 비기능 요구사항

---

## 1. 공통 응답 규격

### 성공 응답

```json
{
  "success": true,
  "data": { ... }
}
```

### 에러 응답

```json
{
  "success": false,
  "error": {
    "code": "YF-001",
    "message": "입력값이 올바르지 않습니다"
  }
}
```

### 에러 코드 체계

| HTTP Status | 코드 | 의미 |
|-------------|------|------|
| 400 | YF-001 | 입력값이 올바르지 않습니다 |
| 401 | YF-002 | 인증이 필요합니다 |
| 403 | YF-003 | 접근 권한이 없습니다 |
| 404 | YF-004 | 리소스를 찾을 수 없습니다 |
| 409 | YF-005 | 이미 존재하는 리소스입니다 |
| 500 | YF-500 | 서버 내부 오류가 발생했습니다 |

## 2. 비기능 요구사항

### 2.1 성능

| 항목 | 목표 |
|------|------|
| 정책 목록 API 응답 시간 | p95 < 300ms |
| 정책 상세 API 응답 시간 | p95 < 200ms |
| 키워드 검색 응답 시간 | p95 < 500ms |
| Q&A 첫 토큰 응답 시간 (TTFT) | < 2s |
| 동시 접속자 수 | MVP 기준 100명 |

### 2.2 보안

| 항목 | 요구사항 |
|------|---------|
| 인증 | JWT 기반, access token 만료 시간 짧게 설정 (30분 이내) |
| 토큰 저장 | refresh token은 서버 DB 저장, 클라이언트는 httpOnly 쿠키 또는 안전한 저장소 사용 |
| API 보호 | 인증 필수 엔드포인트와 공개 엔드포인트 분리 |
| 시크릿 관리 | 환경 변수 기반, 코드에 비밀값 포함 금지 |
| 입력 검증 | Bean Validation 기반 요청 경계 검증 |
| OWASP | SQL Injection, XSS, CSRF 방어 기본 적용 |

### 2.3 안정성

| 항목 | 요구사항 |
|------|---------|
| 가용성 | MVP 단계 99% (월간 약 7시간 허용 다운타임) |
| 데이터 백업 | PostgreSQL 일 1회 백업 |
| LLM 장애 시 | Q&A·적합도 판정 기능 graceful degradation, 정책 탐색은 정상 동작 |
| 에러 처리 | 일관된 에러 응답 구조 (ApiResponse.error) |

### 2.4 확장성

| 항목 | 설계 방향 |
|------|----------|
| 모듈 구조 | DDD 기반 모듈 분리 — 향후 마이크로서비스 전환 가능한 구조 |
| 검색 | 키워드 검색 → 벡터 검색 → 하이브리드 검색 순차 고도화 |
| 수집 | 동기 ingestion → 비동기 이벤트 기반 파이프라인 전환 가능 |
| 적합도 | 규칙 기반 → 룰 엔진 + 감사 메타데이터 확장 가능 |

### 2.5 비용 방어

| 항목 | 방어 장치 |
|------|----------|
| LLM 호출 | source hash 기반 변경 감지, 생성 결과 캐시 |
| 임베딩 | 원문 변경 시에만 재계산 |
| 비로그인 요청 | 비로그인 핫패스에서 고비용 LLM 생성 직접 차단 |
| 가이드 | 사전 생성(pre-generation) 우선, 실시간 생성 최소화 |
