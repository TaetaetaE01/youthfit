# 쉬운 해석 가이드 (Easy Policy Interpretation) — Design Spec

> **버전**: v0.1
> **작성일**: 2026-04-28
> **모듈**: `guide` (백엔드), `PolicyDetailPage` (프론트)
> **선행 결정 회의록**: brainstorming 세션 (2026-04-28)

---

## 1. 목표와 비목표

### 1.1 목표
- 청년 정책 원문에 등장하는 어려운 행정 용어를 **일상어로 풀어**, 사용자가 원문을 처음부터 끝까지 읽지 않아도 자격·혜택·신청 방법을 파악할 수 있게 한다.
- 풀이의 **출처를 항상 시각적으로 검증 가능**하게 만들어, "쉽게 설명하되 지나치게 단순화하지 않는다"는 PRODUCT.md 원칙을 UI 자체로 강제한다.
- 복지로 API의 짧은 구조화 필드만으로도 가치 있는 가이드를 생성하되, 첨부파일 임베딩이 추가되면 **자동으로 풍부해지는** 입력 모델을 갖는다.

### 1.2 비목표
- 첨부파일(PDF/HWP) 추출·임베딩 파이프라인 — `ingestion` 모듈의 별도 spec.
- 사용자의 lazy 트리거(미생성 정책에 대한 "AI 해석 받기" CTA) — 메트릭 보고 v0.x에서 결정.
- 적합도 판정·Q&A — 다른 모듈 책임.
- 법적 효력이 있는 자격 판정 — 가이드는 참고용.

## 2. 사용자 시나리오

페르소나 C (박서연, 23세, 정책 처음 접함)가 정책 상세 페이지에 진입한다. 상단에 한 줄 요약이 보여 정책의 정체를 즉시 파악한다. 스크롤하면 원문 "지원대상" 섹션 위에 같은 주제의 **쉬운 해석**이 불릿으로 정리되어 있다. 의심되면 바로 옆 원문에서 확인할 수 있다. 페이지 하단의 **놓치기 쉬운 점** 카드에서 "월세 60만원 초과 주택은 대상 제외" 같은 함정을 발견하고, 출처 라벨(`지원대상`)을 클릭해 위 페어드 섹션으로 스크롤된 원문에서 검증한다.

## 3. 출력 구조

### 3.1 가이드의 5개 콘텐츠 단위

| # | 단위 | 위치 | 입력 | 비어있을 수 있나 |
|---|------|------|------|-----------------|
| 1 | 한 줄 요약 | 페이지 상단 단독 카드 | `title`, `summary`, `body` | 항상 생성 |
| 2 | 쉬운 지원대상 | `지원대상` 원문 페어 | `supportTarget` | 원문 비면 페어 자체 생략 |
| 3 | 쉬운 선정기준 | `선정기준` 원문 페어 | `selectionCriteria` | 원문 비면 페어 자체 생략 |
| 4 | 쉬운 지원내용 | `지원내용` 원문 페어 | `supportContent` | 원문 비면 페어 자체 생략 |
| 5 | 놓치기 쉬운 점 | 페이지 하단 단독 카드 | 모든 텍스트 결합 | 함정 없으면 카드 미렌더 |

### 3.2 각 단위의 형식·어조 규칙

- **불릿 리스트** (한 줄 요약 제외, 한 줄 요약은 1~2문장 자유 텍스트)
- **명사형/단정형 어조** 유지 — 원문 톤과 일치
- 친근체(`~예요`, `~드려요`, `~요`) **사용 금지**
- 어려운 행정 용어를 **일상어로 치환**하되, 의미 변경 금지
- 환경값(`중위소득 N%` 등)은 가능하면 **연도 기준 환산값을 병기** (예: `중위소득 60% 이하 (2026년 기준 월 약 130만원)`)

## 4. 데이터 모델

### 4.1 Guide 엔티티 (백엔드, 변경)

**Before**
```
guide
├── id
├── policy_id (UNIQUE)
├── summary_html: TEXT
└── source_hash: VARCHAR(64)
```

**After**
```
guide
├── id
├── policy_id (UNIQUE)
├── content: JSONB         ← summary_html 컬럼 대체
└── source_hash: VARCHAR(64)
```

`content` JSON 스키마:
```json
{
  "oneLineSummary": "string",
  "paired": {
    "target":   { "items": ["string", ...] } | null,
    "criteria": { "items": ["string", ...] } | null,
    "content":  { "items": ["string", ...] } | null
  },
  "pitfalls": [
    {
      "text": "string",
      "sourceField": "supportTarget" | "selectionCriteria" | "supportContent" | "body"
    }
  ]
}
```

- `paired.{target|criteria|content}`가 `null`이면 **해당 페어 카드를 렌더하지 않음**.
- `pitfalls`는 빈 배열 허용. 비어있으면 카드 미렌더.
- 도메인 메서드: `hasChanged(newHash)`, `regenerate(content, newHash)`.

### 4.2 마이그레이션
- `guide.summary_html` 컬럼 DROP, `content jsonb NOT NULL` ADD.
- 현재 운영 중인 데이터 없음 (호출부 미존재) → 단순 교체 가능. 별도 백필 불필요.

### 4.3 입력 모델 (LLM에 전달되는 텍스트 번들)

```
[필수] Policy 구조화 필드
  - title, summary, body
  - supportTarget, selectionCriteria, supportContent
  - applyMethods, contact, organization, regionCode, applyStart, applyEnd

[옵션] PolicyDocument 청크 (있으면 결합)
  - 동일 policyId의 청크 전부 (chunkIndex 순)
  - 첨부 추출 파이프라인이 채워 넣기 전엔 빈 상태 → 가이드는 그래도 동작
```

## 5. UI/UX 디자인

### 5.1 PolicyDetailPage 좌측 메인 컬럼 순서 (변경)

```
1. PolicyHeader (제목 + 북마크)            [기존]
2. ✨ 한 줄 요약 카드                       [기존 GuideSummaryCard 자리, 콘텐츠만 교체]
3. 정책 요약 (원문 policy.summary)          [기존, 페어 없음 — 그 역할은 위 #2 "한 줄 요약"이 수행]
4. PolicyMetaSummary                       [기존]
5. 페어 #1: 쉬운 지원대상 + 지원대상 원문    [NEW pairing]
6. 페어 #2: 쉬운 선정기준 + 선정기준 원문    [NEW pairing]
7. 페어 #3: 쉬운 지원내용 + 지원내용 원문    [NEW pairing]
8. ⚠️ 놓치기 쉬운 점 카드                   [NEW]
9. ReferenceSiteSection                    [기존]
10. AttachmentSection                      [기존]
11. 공식 신청 채널                          [기존]
12. Q&A                                    [기존]
```

### 5.2 페어드 섹션 시각 구성

```
┌───────────────────────────────────────┐
│ 🟦 누가 받을 수 있나요 (쉬운 해석)         │
│   • 만 19~34세                          │
│   • 부모와 따로 살고 본인 명의 계약자       │
│   • 본인 소득 월 130만원 이하            │
├───────────────────────────────────────┤
│ ⚪ 지원대상 (원문)                        │
│   만 19세 이상 34세 이하의 무주택 세대주… │
└───────────────────────────────────────┘
```

- 두 박스를 한 컨테이너로 묶어 시각적 연결.
- 쉬운 해석 박스는 **브랜드 색 톤**(연한 indigo 배경), 원문 박스는 **중립 톤**(흰 배경).
- 한 화면에 함께 들어와야 하므로 모바일에선 vertical stack, 데스크탑도 동일(2컬럼 분리하지 않음 — 사이드바 외).

### 5.3 놓치기 쉬운 점 카드

```
┌─ ⚠️ 놓치기 쉬운 점 ──────────────────┐
│                                      │
│ • 월세 60만원 초과 주택은 대상 제외     │
│   └ [지원대상 ↗]                      │
│                                      │
│ • 부모와 같이 살면 자격 없음           │
│   └ [지원대상 ↗]                      │
│                                      │
│ • 신청 후 3개월 내 입주 확인 필요      │
│   └ [지원내용 ↗]                      │
└──────────────────────────────────────┘
```

- 출처 라벨(`[지원대상 ↗]`)은 클릭 가능. 클릭 시 페어드 섹션의 원문 박스로 부드럽게 스크롤(`scrollIntoView({behavior: 'smooth'})`) + 짧은 하이라이트(노란 배경 1.5초).
- `sourceField`가 `body`인 경우 라벨은 `정책 본문`으로 표시, 클릭 시 정책 요약 카드로 스크롤 (가장 가까운 원문 표시 영역).

### 5.4 빈 페어 처리

원문 필드가 비어있으면 **해당 페어 컨테이너를 렌더하지 않는다**. 빈 슬롯 / "정보 없음" 박스 / 회색 placeholder는 두지 않는다 — 출처 1:1 매핑이 페어 레이아웃의 핵심 가치이므로 출처가 없으면 페어가 존재해선 안 된다.

### 5.5 미생성 정책 처리

가이드가 아직 생성 안 된 정책에 진입하면 **상단 한 줄 요약 카드와 모든 페어드 쉬운 해석 박스, 놓치기 쉬운 점 카드를 모두 숨긴다**. 사용자는 원문 섹션과 메타 정보만 보게 된다. 별도 안내 문구·placeholder 없음.

### 5.6 디스클레이머

상단 한 줄 요약 카드 하단에 항상 노출:
> AI가 정리한 해석이에요. 정확한 조건은 아래 원문과 공식 공고에서 확인해주세요.

## 6. LLM 프롬프트 설계

### 6.1 호출 단위
- 가이드 1건 = LLM 호출 1회. 5개 단위를 모두 포함한 단일 JSON 출력.
- OpenAI Structured Outputs (`response_format: { type: "json_schema", schema: ... }`) 사용해 스키마 강제.
- 모델: `gpt-4o-mini` (PRD/ARCHITECTURE.md 기존 결정 유지).

### 6.2 시스템 프롬프트 (요지)

```
너는 한국 청년 정책 가이드를 만드는 보조자다.

원칙:
1. 입력으로 주어진 원문에만 근거하라. 원문에 없는 조건·금액·기한·자격을 추가하지 마라.
2. 원문에 명시된 모든 금액·연령·기간·지역 조건은 풀이에 누락 없이 보존하라.
3. 어려운 행정 용어를 일상어로 치환할 때 의미를 바꾸지 마라.
4. 어조는 원문의 명사형/단정형을 유지하라. "~예요", "~드려요" 같은 친근체 금지.
5. 추정·가정·예시·"~일 수 있어요" 같은 표현 금지.
6. 환경값(중위소득 N% 등)은 가능하면 입력의 referenceYear 기준 환산값을 괄호로 병기하되, 환산값이 입력에 없으면 만들어내지 마라.

출력 단위:
- oneLineSummary: 정책의 정체를 1~2문장으로. 누가/무엇을/어떻게 받는지 핵심만.
- paired.target / criteria / content: 각각 supportTarget / selectionCriteria / supportContent 원문의 풀이.
  원문이 비어있으면 해당 키를 null로 설정.
- pitfalls: 사용자가 원문을 한 번 읽고 놓칠 만한 함정 항목.
  각 항목에 sourceField (어느 원문 필드에서 도출했는지) 라벨 필수.
```

### 6.3 유저 프롬프트 (구성)

```
[정책 메타]
title:        ...
referenceYear: ...
regionCode:   ...
applyStart/applyEnd: ...

[원문 - 구조화 필드]
summary:           ...
body:              ...
supportTarget:     ...
selectionCriteria: ...
supportContent:    ...
applyMethods:      ...

[원문 - 보강(있으면)]
PolicyDocument 청크 1: ...
PolicyDocument 청크 2: ...
...
```

### 6.4 후처리 검증

LLM 응답 수신 후 서버에서 다음을 검증한다. 실패 시 경고 로그(차단 아님, v0에서는 운영 검토용 메트릭).

| 검증 | 내용 |
|------|------|
| 스키마 | JSON 파싱 성공, 필수 키 존재. 실패 시 즉시 재시도 1회 후 폐기 |
| 숫자 보존 | 원문에 등장한 금액·연령·기간 토큰(`\d+만원`, `만\s*\d+세`, `\d+개월`)이 풀이에 보존되는지 — 빠진 토큰 목록을 로그 |
| 출처 매핑 | `pitfalls[].sourceField`가 실제로 비어있지 않은 입력 필드인지 확인 — 빈 필드를 출처로 지목 시 해당 항목 폐기 |
| 친근체 검출 | 풀이 텍스트에 `~예요|~드려요|~해요`가 등장하면 경고 로그 |

후처리 검증 강도는 v0.1에서는 **로깅 위주**, v0.2에서 메트릭 보고 차단/재생성 정책으로 강화.

## 7. 생성 트리거 / 캐시 정책

### 7.1 사전 생성 (메인 경로)

새벽 ingestion 배치(n8n → 내부 ingestion API) 완료 후, 같은 배치 흐름에서 신규/변경 정책에 대해 가이드 생성을 트리거한다. 트리거 위치는 다음 두 가지 중 하나로 plan 단계에서 확정한다.

- (a) `IngestionService`가 정책 upsert 후 같은 트랜잭션 외부에서 `GuideGenerationService.generateGuide` 직접 호출 — 단순, 단 LLM 응답 지연이 ingestion 흐름에 누적
- (b) ingestion은 정책 저장만 하고, 별도 야간 스케줄러가 `Guide.sourceHash` ≠ 입력 hash인 정책을 backfill — 깔끔, 1단계 지연

어느 경로든 다음을 보장한다:
- 가이드 생성 실패는 ingestion 자체를 실패시키지 않는다.
- `Guide.sourceHash` ≠ 입력 텍스트 hash인 경우만 LLM 재호출.
- 동일 hash면 스킵 (PRD 7.5 비용 방어).

- `Guide.sourceHash` ≠ 입력 텍스트 hash인 경우만 LLM 재호출.
- 동일 hash면 스킵 (PRD 7.5 비용 방어).

### 7.2 미생성 정책

조회 시 가이드가 없으면 가이드 관련 모든 카드(한 줄 요약, 페어, 놓치기 쉬운 점)를 숨긴다. 사용자에 별도 안내 없음. lazy 생성 트리거 없음.

### 7.3 운영 모니터링

- 가이드 생성 성공률, LLM 토큰 사용량, sourceHash 캐시 히트율을 메트릭으로 노출.
- 미생성 정책 비율이 5% 초과 시 알림 → ingestion 트리거 로직 점검.

## 8. API 변경

### 8.1 가이드 조회

`GET /api/v1/guides/{policyId}`

**응답 (200 OK)** — 변경:
```json
{
  "success": true,
  "data": {
    "policyId": 1,
    "oneLineSummary": "만 19~34세 청년이 월세 부담을 덜 수 있게…",
    "paired": {
      "target":   { "items": ["만 19~34세", "..."] },
      "criteria": null,
      "content":  { "items": ["매월 최대 20만원", "..."] }
    },
    "pitfalls": [
      { "text": "월세 60만원 초과 주택은 대상 제외", "sourceField": "supportTarget" }
    ],
    "updatedAt": "2026-04-28T03:21:00"
  }
}
```

**응답 (404)** — 가이드 미생성: 프론트는 이를 받으면 가이드 관련 카드 전부 숨김.

### 8.2 가이드 생성 (관리/내부)

`POST /api/v1/guides/generate` 는 유지하되, 기본 트리거 경로는 ingestion 배치다. 이 엔드포인트는 운영자용 수동 트리거로만 사용.

## 9. 워딩 후보

### 9.1 한 줄 요약 카드 헤더
- **이 정책 한눈에** ← 추천
- 한 줄 요약
- 한마디로
- 이 정책은 뭔가요

### 9.2 페어드 쉬운 해석 박스 헤더
- **(원문 섹션명)을 쉬운 말로** — 예: "지원대상을 쉬운 말로"
- 누가 받을 수 있나요 / 어떻게 뽑히나요 / 무엇을 받나요 ← 추천 (질문형이 페르소나 C 진입 친화적)
- 풀어 쓴 (원문 섹션명)

### 9.3 "놓치기 쉬운 점" 카드 헤더
- **놓치기 쉬운 점** ← 가장 직관적
- 꼭 확인하세요
- 헷갈리기 쉬운 조건
- 이건 꼭 짚고 가요
- 사전 체크 포인트
- 잘못 알기 쉬워요
- 미리 확인하세요

**v0.1 추천 조합**:
- 헤더 1: `이 정책 한눈에`
- 헤더 2 (페어): `누가 받을 수 있나요` / `어떻게 뽑히나요` / `무엇을 받나요`
- 헤더 3: `놓치기 쉬운 점`

워딩은 spec 리뷰 단계에서 최종 결정.

## 10. 수용 기준 (Acceptance Criteria)

기능적:
- [ ] 가이드가 생성된 정책 상세 페이지에 한 줄 요약 카드, 페어드 쉬운 해석 박스(원문이 있는 섹션만), 놓치기 쉬운 점 카드가 표시된다.
- [ ] 가이드가 없는 정책에서는 가이드 관련 카드가 모두 숨겨진다.
- [ ] 놓치기 쉬운 점 항목의 출처 라벨 클릭 시 해당 원문 박스로 스크롤되고 1.5초 하이라이트된다.
- [ ] 원문이 비어있는 페어는 렌더링되지 않는다.
- [ ] `Guide.sourceHash`가 일치하면 LLM 재호출하지 않는다.

비기능:
- [ ] 가이드 조회 응답시간 p95 < 200ms (캐시된 JSON 조회).
- [ ] 가이드 생성 성공률 ≥ 95% (스키마 검증 통과 기준).
- [ ] 친근체(`~예요/~드려요/~해요`) 출현율 < 1% (랜덤 샘플 100건 기준).
- [ ] 원문에 있던 금액·연령·기간 토큰의 풀이 보존율 ≥ 90% (자동 검증 메트릭).

## 11. 비범위 / 미해결

- **첨부파일 추출/임베딩 파이프라인** — 별도 spec. 완성되면 가이드 입력이 자동 보강됨(코드 변경 없음).
- **사용자 lazy 트리거** — 미생성 정책 비율 메트릭 보고 v0.x에서 결정.
- **사용자 피드백 메커니즘** ("이 풀이 틀렸어요" 신고) — v1.
- **다국어** — v1+.
- **연 단위 환경값(중위소득 등) 자동 환산** — v0.1에서는 LLM이 입력에 있는 값만 사용. 환산값 자동 계산은 별도 데이터 소스 필요 (v0.x).
