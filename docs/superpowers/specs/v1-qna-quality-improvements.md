## 배경

v0 의미 캐시 도커 검증 단계에서 사용자 테스트 결과, **의미 캐시와 무관한** 두 가지 기존 Q&A 품질 이슈가 발견됨.

### 이슈 1 — "관련 내용 없음" 답변에도 출처가 표시됨

**현상**: 다음 같은 응답이 나오는데 sources 박스에 청크 5개가 그대로 표시됨.

> 해당 정책 원문에 관련 내용이 명시되어 있지 않습니다. 공식 문의처에서 확인하시는 것을 권장합니다.

**근본 원인**:
- `application.yml`의 `relevance-distance-threshold` 기본값이 `0.7`로 매우 느슨함
- RAG 검색 결과 top5 distance가 `0.65~0.69` 정도면 모두 통과
- 통과한 청크들이 LLM에 컨텍스트로 전달되지만, 실제로는 답이 없는 내용
- LLM이 자체 판단으로 "관련 내용 없음"이라고 자작 답변 생성
- `QnaService`는 LLM이 "모른다"고 했는지 알지 못함 → LLM에게 준 청크를 그대로 `sources`로 스트리밍

**잘못된 결과**:
- 사용자 입장에서 "근거 없다"는 답변과 "p.X-Y 출처 5개"가 동시에 표시되어 혼란
- 출처를 클릭해도 답을 보강할 정보가 없음 (애초에 관련 없어서 LLM도 못 답한 청크)

### 이슈 2 — 메타 질문 처리 불가

**현상**: "어떤 정책이야?", "이 정책 개요는?" 같은 정책 자체를 묻는 질문에도 "찾을 수 없다"고 응답.

**근본 원인**:
- `OpenAiQnaClient.SYSTEM_PROMPT`가 "반드시 제공된 컨텍스트에 근거하여 답변하세요"로 LLM을 강하게 제약
- user message에 들어가는 정보는 `정책명: {title}` + `정책 원문 컨텍스트: {chunks}` 만
- 청크는 세부 절차/조건 위주라 정책 개요·목적·범주 정보가 없음
- LLM은 정책명 외에 답할 근거가 없어 "정보 없음"으로 응답

**잘못된 결과**:
- 사용자가 정책에 대해 가장 먼저 묻고 싶은 "이게 뭐야?" 질문이 막힘
- Q&A의 실용성을 크게 떨어뜨림

## 제안 방향

### 이슈 1 해결책

**A. `relevance-distance-threshold`를 0.4~0.5로 조이기 (권장)**
- 진짜 관련 있는 청크만 LLM에 전달
- 미관련이면 `QnaService.processQuestion`이 자체적으로 `NO_RELEVANT_MESSAGE` 흐름으로 처리 → `sendSourcesEvent(emitter, List.of())`로 빈 출처 전송
- 부수 효과: LLM 호출이 줄어 비용 절감
- 위험: 너무 빡빡하면 정상 질문도 거절 → 운영 데이터로 0.4~0.55 사이 튜닝

**B. LLM 응답에서 "모른다" 패턴 감지 후 sources 제거 (보조)**
- 문구 매칭("관련 내용", "찾을 수 없", "명시되어 있지 않") 으로 일치 시 `sources = List.of()`
- 단점: 표현 변형에 취약, 거짓 양성/음성 모두 가능
- A의 보조 수단으로만 가치 있음

**C. 시스템 프롬프트에 명시적 "NO_ANSWER" 토큰 합의 (보조)**
- "답을 찾을 수 없으면 정확히 `NO_ANSWER` 토큰만 반환"하도록 지시
- 토큰 보이면 `NO_RELEVANT_MESSAGE` 치환 + sources 비움
- 단점: LLM이 합의 토큰을 항상 지키진 않음 (확률적)

권장: **A를 메인으로**, 안전망으로 B를 함께 적용. 임계값은 0.5 정도로 시작.

### 이슈 2 해결책

**A. 정책 메타데이터를 user message에 포함 (권장)**
- 현재:
  ```
  정책명: 청년내일저축계좌

  정책 원문 컨텍스트:
  [청크 0] ...
  ```
- 변경:
  ```
  정책명: 청년내일저축계좌
  정책 분야: {policy.category}
  정책 요약: {policy.summary}
  지원 대상 요약: {policy.targetSummary}
  지원 금액 요약: {policy.benefitSummary}

  정책 원문 컨텍스트:
  [청크 0] ...
  ```
- 시스템 프롬프트에 "정책 메타데이터와 원문 컨텍스트를 모두 활용해 답하세요"로 완화
- `Policy` 엔티티 필드 검토 필요 (어떤 메타 필드가 이미 있는지)

**B. 메타 질문 분기 (보조)**
- 패턴 매칭으로 "어떤 정책", "이 정책 뭐", "개요", "소개", "요약" 류 검출
- 검출 시 별도의 metadata-only 프롬프트로 LLM 호출 (또는 RAG 우회)
- 단점: 분기 룰 유지 부담

권장: **A**. 모든 질문에 메타데이터를 함께 주는 게 단순하고 효과 큼.

## 영향 범위

| 영역 | 변경 |
|---|---|
| `application.yml` | `relevance-distance-threshold` 기본값 0.7 → 0.5 |
| `QnaService.processQuestion` | (이슈 1-B 적용 시) LLM 답변 패턴 매칭 후 sources 처리 분기 |
| `OpenAiQnaClient.SYSTEM_PROMPT` | (이슈 2-A 적용 시) "메타데이터와 원문 모두 활용" 으로 완화 |
| `OpenAiQnaClient.generateAnswer` | user message 빌더에 정책 메타데이터 추가 |
| `Policy` 엔티티 | 메타 필드 확인. 없으면 추가 필요 (별도 작업) |
| 테스트 | `QnaServiceTest`에 메타 질문 / 미관련 청크 시나리오 추가 |

## 검증 방법

이슈 1:
- 정책 상세 텍스트에 분명히 없는 정보 질문 (예: 개인적 의견, 정책과 무관한 주제)
- 기대: 답변에 `NO_RELEVANT_MESSAGE` + sources 비어 있음

이슈 2:
- "어떤 정책이야?", "이 정책 뭐야?", "정책 개요는?" 등 5~10개 변형
- 기대: 정책 분야/요약 기반으로 합리적 응답 + 의도된 출처 (메타데이터 출처라면 별도 표시)

## 그라쥬에이션 기준

이 작업을 본격 시작할 트리거:
- 사용자 또는 운영팀에서 "출처는 있는데 답이 없다" / "정책이 뭔지도 모른다" 컴플레인 누적
- v0 의미 캐시 출시 후 1~2주 운영 데이터로 두 이슈의 빈도 확인
- 만약 빈도 높으면 우선순위 상향

## 사전 작업

작업 진입 시 점검 항목:
- `Policy` 엔티티의 메타 필드 (요약/카테고리 등) 보유 여부 확인
- 누락 시 ingestion 파이프라인에서 보강 가능한지 확인 (별도 작업 가능성)
- `relevance-distance-threshold` 분포 측정 — 운영 로그에서 통과 청크의 distance 분포 보고 0.5 적정성 검증
