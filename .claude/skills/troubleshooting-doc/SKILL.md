# 트러블슈팅·의사결정 문서화 스킬

## 1. 무엇을 만드는가

Claude Code 와 함께한 한 세션 동안 발생한 **트러블슈팅** 과 **기술적 의사결정** 을
한 장의 마크다운 문서로 남긴다. 다음 두 가지 정보가 같은 문서에 공존한다.

- **트러블슈팅**: 어떤 문제(상황) → 왜 그랬나(원인) → 어떻게 풀었나(해결) → 어떻게 확인했나(결과)
- **기술적 의사결정**: 어떤 후보들이 있었나 → 무엇을 선택했나 → 왜 선택했나 → 무엇을 포기/유예했나

둘 다 있을 수도 있고 하나만 있을 수도 있다. 해당 섹션이 비면 비운 채로 두지 말고,
"이 작업에는 의사결정 분기가 없었다" 정도로 명시한다 — 미래의 독자가 "안 적힌 건지, 정말 없었던 건지"
판단하지 못하면 문서 가치가 떨어진다.

## 2. 저장 위치와 네이밍

### 위치

```
docs/troubleshooting/YYYY-MM-DD-<slug>-trouble.md
```

`docs/superpowers/` (brainstorm → spec → plan → operations 사이클 전용 영역) 안이 아니라
`docs/` 바로 아래에 평평하게 둔다. 이유: 트러블슈팅·의사결정 기록은 사이클의 산출물이 아니라
**그때 무슨 일이 있었는지의 사후 기록**이라 성격이 다르고, 향후 누구나(슈퍼파워스 흐름과
무관하게) 참고할 수 있어야 하기 때문이다.

`docs/superpowers/operations/` 의 런북과도 분리한다. 런북은 "지금 운영 중에 다시 봐야 할 절차"
이고, 트러블슈팅 문서는 "그때 무슨 일이 있었는지의 기록"이라 결이 다르다.

디렉토리가 없으면 만든다.

### 파일명 규칙

- `YYYY-MM-DD`: 문서를 처음 작성한 날짜. 사건이 일어난 날이 아닌 **기록한 날** 기준.
  이후 수정해도 그대로 둔다 (시점 추적 용도).
- `<slug>`: 케밥 케이스 영문 슬러그, 사건의 핵심을 짧게 (3~6 단어).
- 접미사 `-trouble.md` 로 끝내 다른 문서 종류와 구분.

### 같은 날 여러 건이 있을 때

같은 날짜에 둘 이상의 문서를 만들면 **시작 순서대로** 2자리 시퀀스 번호를 날짜 뒤에 붙인다.

```
YYYY-MM-DD-NN-<slug>-trouble.md
```

- `NN`: `01` 부터 시작.
- 하루에 1건뿐이면 시퀀스 생략.
- 2건째가 추가되는 순간 첫 번째 파일에도 `01` 을 소급 적용해 일관성을 맞춘다.

### 예시

```
2026-04-30-qna-securitycontext-virtual-thread-trouble.md   # 그날 1건뿐인 경우
2026-04-30-01-qna-cache-invalidation-trouble.md            # 같은 날 2건째 등장 시
2026-04-30-02-qna-sse-frame-mismatch-trouble.md
```

## 3. 문서 구조 — 표준 템플릿

다음 템플릿을 그대로 사용한다. 빈 섹션은 `_(해당 없음)_` 으로 표시한다.

```markdown
# <한국어 제목 — "무엇을 어쨌다" 가 한 줄에 보이게>

- 작성일: YYYY-MM-DD
- 작성자: <git user.name>
- 관련 커밋: `<sha>` (`<커밋 메시지 한 줄>`)
- 관련 PR: #<번호> (있으면)
- 관련 모듈: <백엔드 모듈명 또는 프론트엔드 영역>

## 한 줄 요약

> 어떤 문제를 어떻게 풀었고 어떤 결정을 내렸는지 두세 줄 안에.

## 1. 상황 (Context)

- 어떤 작업을 하던 중이었나
- 무엇이 잘못 보였나 (재현 조건, 에러 로그, 잘못된 동작의 구체적 묘사)
- 영향 범위 (어떤 사용자/요청/환경에서)

## 2. 원인 (Root Cause)

- 표면 증상이 아니라 **왜** 그런 동작이 나왔는지
- 관련된 코드 경로 (`패키지/파일명:라인`)
- 잘못된 가정이나 누락된 전제

## 3. 고려한 대안 (Alternatives)

여러 해결 방향을 검토했다면 여기에 정리한다. 단일 해결만 고려했다면 본 섹션은 `_(단일 해결, 대안 검토 없음)_` 로.

| 대안 | 장점 | 단점 / 채택 안 한 이유 |
|---|---|---|
| A. <설명> | ... | ... |
| B. <설명> | ... | ... |
| C. <설명> | ... | ... |

## 4. 선택과 이유 (Decision)

- 채택한 대안: <위 표의 항목>
- **결정의 핵심 근거**: (예: 운영 부담 / 응답 지연 / 보안 모델 일관성 / 변경 범위 / 가역성)
- 트레이드오프로 받아들인 것: 이 선택이 포기한 것은 무엇인가
- 가역성: 이 결정을 나중에 뒤집기 쉬운가? 어떤 신호가 보이면 재검토할 것인가?

## 5. 해결 (Solution)

- 실제로 적용한 변경 (코드/설정/배포 단계)
- 변경된 파일·심볼 (`경로:심볼`)
- 마이그레이션·환경변수·캐시 무효화 같은 **반복 불가능한 부수 효과** 가 있다면 명시

## 6. 검증 (Result)

- 어떻게 고쳐졌음을 확인했나 (재현 시나리오, 로그, 테스트, 스크린샷)
- 회귀 위험과 모니터링 포인트
- 측정 가능한 지표 (응답 시간, 에러율, 캐시 히트율 등) 가 있다면 before/after

## 7. 후속 / 미결 (Follow-ups)

- 본 작업에서 의도적으로 뒤로 미룬 항목
- 재발 방지를 위해 만들어둘 만한 가드레일 (테스트, 어설션, 모니터링)
- 별도 spec/plan 으로 옮길 가치가 있는 항목

## 8. 참고 (References)

- 관련 spec/plan/runbook 경로
- 외부 문서·이슈 링크
- 대화에서 결정적이었던 사용자 메시지 인용 (선택)
```

## 4. 작성 절차

1. **세션 컨텍스트 수집** — 이미 진행된 대화에서 다음을 추출한다.
   - 사용자가 처음에 보고한 증상
   - 시도했던 접근들과 각각의 결과
   - 사용자가 명시적으로 받아들이거나 거부한 대안
   - 최종 적용된 코드 변경

2. **관련 git 정보 확보**

   ```bash
   git log -n 5 --oneline
   git show --stat <sha>
   ```

   해당 작업에 연결된 커밋이 있으면 SHA 와 메시지를 메타에 넣는다. 아직 커밋 전이라면
   "관련 커밋: _(미커밋, 워킹 트리 변경)_" 로 표기.

3. **빈 섹션 판단** — 트러블슈팅이 아니라 순수 의사결정만 있었던 세션이라면
   `1. 상황 / 2. 원인` 은 짧게, `3. 대안 / 4. 선택과 이유` 를 본체로.
   반대로 단순 버그 수정이라 대안 검토가 없었던 세션이라면
   `3. 고려한 대안` 은 `_(단일 해결, 대안 검토 없음)_` 로 명시한다.

4. **파일 작성** — `docs/troubleshooting/` 디렉토리가 없으면 생성하고
   템플릿대로 채운다.

5. **다른 README 는 손대지 않는다** — `docs/superpowers/README.md` 는 슈퍼파워스 사이클
   문서만 다루는 별도 영역이라 트러블슈팅 디렉토리와 무관하다. 기본은 어떤 README 도 갱신하지
   않는다. `docs/troubleshooting/` 자체에 README 가 필요하다고 사용자가 별도로 지시할 때만
   추가한다.

6. **마지막으로 문서 경로를 사용자에게 알린다** — 어디에 만들었는지 한 줄로 보고하고,
   필요하면 커밋 메시지(`docs(trouble): <slug>` 또는 `docs: <slug> 트러블슈팅 노트`)를
   제안한다. 커밋은 사용자가 명시적으로 요청하지 않으면 자동으로 만들지 않는다.

## 5. 작성 원칙

### 표면 증상이 아니라 원인을 적는다

"X 가 안 됐다 → Y 로 고쳤다" 는 git diff 가 이미 한다. 문서가 가치 있는 부분은
"왜 X 가 그런 식으로 동작했는가" 다. 추측이면 추측이라고 표시하고, 검증 방법까지 적는다.

### 미래의 독자를 가정한다

3 개월 뒤 이 문서를 처음 보는 팀원/자기 자신이 다음 의사결정을 내릴 때 충분한 맥락이
들어 있어야 한다. 코드를 다시 안 봐도 흐름이 잡혀야 좋다.

### 인용은 정확히

코드 라인을 인용하면 `경로/파일.kt:42` 처럼 라인 번호까지. 임계값·환경변수 이름·SQL 은
원문 그대로. 사람의 발언을 인용한다면 의역하지 말고 원문을 옮긴다.

### 길이는 사건에 비례한다

작은 버그면 한 페이지, 큰 마이그레이션이면 길어도 된다. 빈 섹션을 채우려고 늘리지 말고,
할 말이 없는 섹션은 솔직하게 짧게 처리한다.

### 비밀값·개인정보는 넣지 않는다

API 키, 토큰, 사용자 식별 정보는 절대 본문에 적지 않는다. 환경변수 이름까지만.

## 6. 예시 (참고용)

다음은 가상의 트러블슈팅 한 건의 한 줄 요약·상황·원인 부분만 발췌한 예시.

```markdown
# Q&A SecurityContext 가 virtual-thread 로 전파되지 않던 문제

- 작성일: 2026-04-30
- 관련 커밋: `7a537cb` (fix(qna): SecurityContext 를 virtual-thread executor 로 전파)
- 관련 모듈: backend/qna, backend/auth

## 한 줄 요약
> Q&A 스트리밍을 virtual-thread executor 로 분리하면서 SecurityContext 가 전파되지 않아
> /api/qna 진입은 통과하지만 그 안에서 호출되는 사용자별 정책 평가가 익명 사용자로
> 떨어지던 문제. SecurityContext-aware 한 executor 래퍼를 추가해 해결.

## 1. 상황
- ... (생략)
```

전체 채워진 예시가 필요하면 사용자에게 "기존 트러블슈팅 문서 하나 보여드릴까요?" 라고 묻고
실제 작성 시점에 한 건을 함께 만든다.

## 7. 트리거되지 않아야 하는 경우

- "오늘 한 일 요약해줘" — 단순 작업 로그/일지 요청은 이 스킬이 아니다.
- "이 코드 리뷰해줘" — 리뷰 스킬을 쓴다.
- "PR 본문 작성해줘" — `create-pr` 스킬을 쓴다.
- "운영 중 모니터링 가이드 만들어줘" — `docs/superpowers/operations/` 의 런북이지 트러블슈팅 문서가 아니다.
- 트러블슈팅이 실제로 일어나지 않은 일반적 글쓰기 — 이 스킬을 쓰지 않는다.

판단이 애매하면 사용자에게 "트러블슈팅·의사결정 문서로 정리할까요, 아니면 다른 형식이 좋을까요?"
한 번 묻고 진행한다.
